# AIMAOS Data Status Center Refactor

작성일: 2026-06-11

## 1. 변경 이유

AIMAOS의 핵심 목표는 광고주가 파일을 직접 다운로드하거나 업로드하지 않아도 최신 데이터 기반 분석을 보는 것입니다.

기존 구조는 로그인 후 바로 `오늘 해야 할 일`을 보여주는 방식이었습니다. 이 경우 데이터가 1일 전, 3일 전, 1주일 전이어도 사용자는 최신 분석으로 오해할 수 있습니다.

따라서 첫 화면을 `Data Status Center`로 바꾸고, 사용자가 먼저 데이터 최신성과 매체 연결 상태를 확인한 뒤 오늘 해야 할 일과 보고서를 보도록 구조를 변경했습니다.

## 2. 기존 구조와 충돌하는 부분

| 항목 | 충돌 내용 | 처리 방식 |
| --- | --- | --- |
| 첫 메뉴 | 기존 첫 메뉴는 오늘 해야 할 일 | 첫 메뉴를 Data Status Center로 변경 |
| 데이터 상태 표시 | 기존에는 사이드바 하단 작은 카드 중심 | 독립 화면으로 승격 |
| 메뉴 제한 | 기존에는 데이터 상태와 무관하게 진입 | 연결 안됨일 때만 일부 메뉴 안내 화면 표시 |
| 분석 신뢰도 | 오래된 데이터에서도 분석이 바로 보임 | 주의/오래됨 배너로 최신화 권장 |

## 3. 수정 영향 범위

수정 파일:

- `aimaos/app/streamlit_app.py`

수정하지 않은 영역:

- Rule Engine
- 광고 성과 계산 로직
- KPI 계산 로직
- 보고서 생성 로직
- 컬럼 매핑 로직
- 데이터 표준화 로직
- CSV/Excel 업로드 로직

이번 변경은 데이터 최신성 관리 UI와 상태 관리만 다룹니다.

## 4. 필요한 DB 필드

현재 구현은 기존 파일과 POC 증거 파일의 수정 시각을 기준으로 동작합니다. 실제 운영 DB로 확장할 때 필요한 필드는 아래와 같습니다.

| 필드 | 설명 |
| --- | --- |
| advertiser_id | 광고주 식별자 |
| media | 매체명 |
| connection_status | 연결 완료, 연결 필요, 인증 만료, 오류 |
| last_collection_at | 마지막 데이터 수집 시각 |
| last_analysis_at | 마지막 분석 시각 |
| last_success_at | 마지막 성공 시각 |
| last_failure_at | 마지막 실패 시각 |
| last_failure_reason | 실패 원인 |
| success_count_30d | 최근 30일 성공 횟수 |
| failure_count_30d | 최근 30일 실패 횟수 |
| success_rate_30d | 최근 30일 수집 성공률 |
| auth_status | 정상, 만료, 재인증 필요 |
| data_source_type | API, 반자동 브라우저, 수동 업로드 |

## 5. 상태 정의

| 상태 | 기준 | 색상 |
| --- | --- | --- |
| 최신 | 24시간 이내 | 녹색 |
| 주의 | 24~72시간 | 노란색 |
| 오래됨 | 72시간 초과 | 빨간색 |
| 연결 안됨 | 데이터 없음, 수집 실패, 계정 미연결 | 회색 |

## 6. 변경 구조

변경 전:

```text
로그인
→ 오늘 해야 할 일
→ 운영 현황
→ 보고서
```

변경 후:

```text
로그인
→ Data Status Center
→ 최신 데이터 수집
→ 오늘 해야 할 일
→ 운영 현황
→ 보고서
```

## 7. 화면 구조

`Data Status Center` 화면에 아래 항목을 추가했습니다.

1. 마지막 데이터 수집 시각
2. 마지막 분석 시각
3. 전체 상태
4. 큰 `최신 데이터 수집` 버튼
5. 매체별 데이터 상태
6. 광고주 연결 상태
7. 최근 30일 수집 성공률

## 8. UI 변경 내역

### 사이드바

- 첫 메뉴를 `Data Status Center`로 변경
- `오늘 해야 할 일`은 두 번째 메뉴로 이동
- 사이드바 데이터 상태도 동일한 상태 모델을 사용하도록 변경
- Data Status Center 배지는 최신 상태가 아닌 매체 수를 표시

### 본문

- 첫 화면에 데이터 최신성 카드 3개 표시
- 매체별 상태 테이블 추가
- 상태별 색상 표시 추가
- 주의/오래됨/연결 안됨 배너 추가
- `최신 데이터 수집` 버튼을 가장 크게 배치

## 9. 메뉴 진입 정책

| 전체 상태 | 처리 |
| --- | --- |
| 최신 | 모든 메뉴 정상 표시 |
| 주의 | 노란 배너 표시 후 기존 데이터로 계속 보기 가능 |
| 오래됨 | 빨간 배너와 최신 데이터 수집/기존 데이터로 계속 보기 카드 표시 |
| 연결 안됨 | 오늘 해야 할 일, 운영 현황, 보고서 센터에서 데이터 연결 필요 안내 |

현재 화면에서는 전체 상태가 `주의`로 표시됩니다. 이유는 네이버는 최신이지만 G마켓/옥션/11번가/쿠팡은 실제 데이터 다운로드가 아직 검증되지 않았기 때문입니다.

## 10. 구현 결과

현재 Data Status Center 기준:

| 매체 | 연결 상태 | 데이터 상태 |
| --- | --- | --- |
| 네이버 | 연결 완료 | 최신 |
| G마켓 | 광고센터 진입 완료 | 연결 안됨 |
| 옥션 | 광고센터 진입 완료 | 연결 안됨 |
| 11번가 | 연결 필요 | 연결 안됨 |
| 쿠팡 | 연결 필요 | 연결 안됨 |

G마켓/옥션은 광고센터 진입은 성공했지만 실제 다운로드 파일이 없으므로 최신으로 표시하지 않습니다. 이 기준은 분석 신뢰도를 지키기 위한 보수적 판단입니다.

## 11. 작동 점검

실행한 점검:

```powershell
.\.venv\Scripts\python.exe -m compileall aimaos\app\streamlit_app.py
```

결과: 성공.

브라우저 점검:

- 첫 화면이 Data Status Center로 표시됨
- 사이드바 첫 메뉴가 Data Status Center로 변경됨
- 오늘 해야 할 일 메뉴 진입 시 최신 데이터 수집 권장 배너 표시됨
- 오늘 해야 할 일 본문은 유지됨

## 12. 남은 과제

1. `최신 데이터 수집` 버튼을 실제 수집 실행 큐와 연결해야 합니다.
2. G마켓/옥션 다운로드 파일 생성 성공 여부를 실제 광고 데이터 있는 계정으로 검증해야 합니다.
3. 수집 성공률은 현재 구조상 수동 문서/파일 기반이며, 운영 DB로 이전해야 합니다.
4. 로그인 만료, 인증 오류, 매체 장애 사유를 별도 상태값으로 분리해야 합니다.
5. 매체별 상태 이력을 DB에 저장해야 합니다.

## 13. 최종 판단

이번 리팩토링으로 AIMAOS의 첫 화면은 분석 결과가 아니라 데이터 신뢰도를 먼저 확인하는 구조로 바뀌었습니다.

이는 AIMAOS가 단순 광고 분석 툴이 아니라 광고 운영 OS가 되기 위한 필수 전환입니다.

현재 구현은 UI/상태 관리 단계이며, 다음 단계는 `최신 데이터 수집` 버튼을 실제 네이버 API 수집과 G마켓/옥션 반자동 수집 실행 흐름에 연결하는 것입니다.
## 14. 2026-06-11 Night Build 반영

이번 작업에서 Data Status Center를 데이터 최신성 관제 화면으로 보강했습니다.

추가 표시 항목:

- 마지막 성공 시각
- 마지막 실패 시각
- 실패 원인
- 매체별 마지막 확인 시각
- 전체 저장 용량
- 삭제 예정 파일 수
- 삭제 예정 용량
- 다음 정리 상태

G마켓/옥션은 광고센터 진입 성공만으로 최신 데이터 수집 성공으로 보지 않습니다. 실제 리포트 다운로드 파일이 확인되어야 수집 성공으로 판단합니다.

추가된 운영 명령:

```powershell
python -m aimaos.storage.collection_log
python -m aimaos.storage.retention_cleanup --dry-run
python -m aimaos.storage.retention_cleanup --apply
```

이번 변경은 Rule Engine, 분석 계산 로직, 보고서 생성 로직을 수정하지 않고 데이터 최신성 상태 UI와 저장 관리 구조만 보강했습니다.
