# NAVER API 및 반자동 리포트 수집 전략

작성일: 2026-06-09

## 1. 문서 목적

이 문서는 AIMAOS가 매일 수동 파일 다운로드와 업로드에 의존하지 않도록 하기 위한 데이터 연동 구조를 검토합니다.

중요한 전제는 다음과 같습니다.

- 오토다비드의 자동입찰 기능을 따라 하지 않는다.
- 참고할 것은 "최초 연결 이후 웹 기반으로 반복 작업을 처리하는 운영 구조"이다.
- AIMAOS의 목적은 자동입찰이 아니라 광고 데이터 수집, 분석, 오늘 해야 할 일 생성, 보고서 생성이다.
- 기존 광고 분석 계산 로직, Rule Engine, 보고서 생성 구조는 변경하지 않는다.
- 보안 우회, 캡차 우회, 2차 인증 우회는 하지 않는다.

## 2. 데이터 수집 성공 기준

AIMAOS의 데이터 수집 기능은 "기술적으로 가능하다"가 아니라 "광고주와 운영자가 계속 사용할 수 있다"를 기준으로 판단한다.

성공 기준은 아래와 같다.

| 기준 | 성공 조건 |
| --- | --- |
| 파일 다운로드 | 광고주가 직접 광고센터에서 파일을 다운로드하지 않는다. |
| 파일 업로드 | 광고주가 매일 AIMAOS에 파일을 직접 업로드하지 않는다. |
| 매일 로그인 | 광고주가 데이터 수집을 위해 매일 로그인하지 않는다. |
| 사용자 행동 수 | 광고주가 해야 하는 행동은 가능하면 1클릭 이하로 줄인다. |
| 이탈 방지 | 데이터 수집이 번거로워 AIMAOS 사용을 포기하면 실패로 본다. |
| 수집 성공률 | 운영 목표는 95% 이상이다. |
| 실패 설명 | 수집 실패 시 원인을 사용자가 이해할 수 있게 설명한다. |

### 2.1 수집 실패 시 사용자 안내 원칙

데이터 수집 실패 메시지는 개발자 오류가 아니라 운영자가 바로 행동할 수 있는 문장이어야 한다.

예시:

| 실패 상황 | 사용자 안내 |
| --- | --- |
| API 키 오류 | 네이버 검색광고 연결 정보가 맞지 않습니다. Customer ID, Access License, Secret Key를 다시 확인해 주세요. |
| 권한 부족 | 현재 계정으로 이 광고주의 데이터를 조회할 권한이 없습니다. 대행사 권한 또는 광고주 계정 연결 상태를 확인해 주세요. |
| 세션 만료 | 광고센터 로그인 시간이 만료되었습니다. 연결을 다시 확인해 주세요. |
| 2차 인증 필요 | 광고센터에서 추가 인증이 필요합니다. 인증 완료 후 다시 수집을 실행해 주세요. |
| 리포트 다운로드 실패 | 광고센터 리포트 파일을 받지 못했습니다. 기간 선택 또는 광고센터 상태를 확인해 주세요. |
| 컬럼 구조 변경 | 리포트 형식이 이전과 달라졌습니다. 원본 파일 진단으로 컬럼 구조를 확인해 주세요. |

### 2.2 수집 방식별 합격 기준

| 수집 방식 | 1차 합격 기준 | v1 운영 목표 |
| --- | --- | --- |
| 네이버 API 자동 수집 | 최초 연결 후 전일 데이터 자동 수집 성공 | 광고주 추가 행동 없이 매일 자동 수집 |
| G마켓/옥션 반자동 수집 | 사용자가 직접 로그인한 세션에서 리포트 다운로드 자동화 성공 | 로그인 빈도 최소화, 실패 원인 안내 |
| 11번가 반자동 수집 | 리포트 페이지 이동, 기간 선택, 파일 다운로드 자동화 성공 | 광고주가 파일을 직접 다루지 않는 구조 |

### 3.1 오토다비드형 UX에서 참고할 점

오토다비드 사용 안내에는 대행사 마케터를 위한 다중 접속, API 연동, 계정 리스트 확인, 광고그룹 연결, 동기화, 최근 7일 광고지표 표시 흐름이 포함되어 있다.

| 참고 요소 | AIMAOS 적용 방향 |
| --- | --- |
| 최초 API 연동 | 광고주 또는 대행사가 API 정보를 한 번 입력 |
| 계정 리스트 확인 | 연결된 광고주 계정과 매체 상태를 AIMAOS에서 확인 |
| 웹 기반 운영 | 별도 엑셀 수동 업로드 없이 웹에서 수집 상태 확인 |

## 4. 네이버 검색광고 API 연동 구조

### 4.1 공식 확인 사항

네이버 검색광고 API 공식 문서 기준 서비스 URL은 다음이다.

```text
https://api.searchad.naver.com
```

API License와 Secret Key는 네이버 검색광고 광고주센터 가입 후 관리 화면의 API Manager에서 생성하는 흐름이다.

네이버 검색광고 API는 캠페인, 광고그룹, 키워드 조회 계열과 통계 조회 계열을 제공한다. 공식 릴리즈 문서에는 아래 계열이 확인된다.

- 캠페인 조회: `GET /ncc/campaigns`
- 광고그룹 조회: `GET /ncc/adgroups`
- 키워드 조회: `GET /ncc/keywords`
- 통계 조회: `GET /stats`
- 리포트 다운로드: MASTER-REPORT, STAT-REPORT 다운로드 URL 변경 공지 존재
- 쇼핑검색광고 검색어/전환 상세 리포트 항목 제공 이력 존재

### 4.2 연결 시 입력받을 값

| 입력 항목 | 설명 | 저장 방식 |
| --- | --- | --- |
| 계정 별칭 | 운영자가 알아보기 위한 이름 | 평문 저장 가능 |
| 광고주명 | AIMAOS 내부 광고주명 | 평문 저장 가능 |
| Customer ID | 네이버 검색광고 광고주 번호 | 민감 정보로 분류 |
| Access License 또는 API Key | API 호출 식별 값 | 암호화 저장 |
| Secret Key | 서명 생성용 비밀키 | 반드시 암호화 저장 |
| 대행사/광고주 구분 | 다계정 운영 여부 판단 | 평문 저장 가능 |
| 연결 상태 | 정상, 실패, 권한 부족, 만료 | 평문 저장 가능 |

### 4.3 수집 가능 데이터 후보

### 4.4 네이버 API 수집 흐름

```text
1. 운영자가 AIMAOS에서 네이버 연결하기 클릭
2. Customer ID, Access License, Secret Key, 계정 별칭 입력
3. AIMAOS가 읽기 테스트 호출
4. 연결 성공 시 계정 레지스트리에 저장
5. 매일 새벽 전일 성과 수집
6. 원본 JSON/CSV 저장
7. AIMAOS 표준 컬럼으로 변환
8. 기존 분석 파이프라인 실행
9. 오늘 해야 할 일 엔진 실행
10. 보고서 센터에 자동 반영
```

### 4.5 네이버 API 저장 구조 초안

```text
data/connections/naver_searchad_connections
data/raw/naver_searchad/YYYY-MM-DD/{customer_id}/
data/processed/naver_searchad/YYYY-MM-DD/{customer_id}/
data/reports/{advertiser}/{YYYY-MM-DD}/
```

연결 테이블 후보:

| 필드 | 설명 |
| --- | --- |
| connection_id | 내부 연결 ID |
| organization_id | 협회 또는 대행사 조직 ID |
| advertiser_id | 광고주 ID |
| account_alias | 계정 별칭 |
| media | NAVER_SEARCH_AD |
| customer_id | 네이버 Customer ID |
| api_license_encrypted | 암호화된 Access License |
| secret_key_encrypted | 암호화된 Secret Key |
| account_type | agency 또는 advertiser |
| sync_status | 정상, 실패, 권한 부족, 점검 필요 |
| last_sync_at | 마지막 수집 시각 |
| created_at | 연결 생성 시각 |

```text
  보고서 생성 가능
```

### 5.2 대행사 다계정 UX

대행사 계정은 여러 광고주를 관리할 가능성이 있으므로 AIMAOS에서는 아래 구조가 필요하다.

| 화면 | 목적 |
| --- | --- |
| 조직 관리 | 대행사 또는 협회 단위 관리 |
| 광고주 관리 | 광고주별 계정과 매체 연결 상태 관리 |
| 계정 별칭 | 동일 매체 다계정 구분 |
| 연결 상태 | 정상/권한 부족/최근 수집 실패 표시 |
| 동기화 | 계정 변경 사항 수동 재확인 |

네이버 API에서 대행사 계정 하나로 모든 광고주 데이터가 조회되는지는 실제 대행사 계정으로 1차 확인이 필요하다. 불가능하면 광고주별 Customer ID/API Key를 개별 등록해야 한다.

## 6. API 미지원 또는 제한 매체 대응

대상:

- G마켓
- 옥션
- 11번가

### 6.1 공식 API 가능성

#### G마켓/옥션

G마켓/옥션은 ESM Trading API가 존재한다. 공식 API 가이드 기준 상품, 주문/배송, 클레임, 정산조회, CS, 서비스 API가 구분되어 있고, ESM+ 마스터 ID와 JWT 기반 인증 구조를 사용한다.

다만 현재 확인된 범위에서 "광고센터 성과 리포트 자동 수집" 전용 API는 별도 검증이 필요하다. 판매/정산 API가 있다고 해서 광고 성과 리포트 API가 있다고 단정하면 안 된다.

#### 11번가

11번가는 OPEN API CENTER를 통해 API KEY 발급 흐름이 존재한다. 다만 현재 확인된 공개 정보는 판매자 운영 API 중심이며, 광고센터 성과 리포트 API는 별도 확인이 필요하다.

### 6.2 결론

| 매체 | 공식 API 우선 검토 | 광고 리포트 API 판단 | AIMAOS 1차 전략 |
| --- | --- | --- | --- |
| 네이버 검색광고 | 가능 | 가능성 높음 | API 자동 수집 |
| G마켓 | ESM API 존재 | 별도 검증 필요 | 반자동 리포트 수집 |
| 옥션 | ESM API 존재 | 별도 검증 필요 | 반자동 리포트 수집 |
| 11번가 | OPEN API 존재 | 별도 검증 필요 | 반자동 리포트 수집 |

## 7. G마켓/옥션/11번가 반자동 수집 구조

### 7.1 목표

광고주가 매일 광고센터에 들어가 기간을 선택하고 파일을 내려받아 AIMAOS에 업로드하는 일을 줄인다.

반자동 수집은 사용자가 직접 로그인하고, AIMAOS가 로그인 이후의 반복 작업만 돕는 구조다.

### 7.2 사용자 흐름

```text
채널 관리
  G마켓/옥션/11번가
    연결하기
      안내 확인
      브라우저 세션 열기
      사용자가 직접 로그인
      캡차/2차 인증은 사용자가 직접 처리
      AIMAOS가 리포트 페이지 이동
      기간 선택
      Excel/CSV 다운로드
      다운로드 파일 감지
      기존 분석 파이프라인 연결
```

### 7.3 수집 방식 후보

#### 방식 A: 서버 제어 브라우저

```text
AIMAOS 서버가 Playwright 브라우저 세션 생성
사용자가 해당 세션에서 직접 로그인
서버가 다운로드 파일을 임시 저장
분석 파이프라인으로 연결
```

장점:

- 웹 기반으로 동작 가능
- 다운로드 파일 감지가 쉽다
- 사용자가 매일 파일을 직접 업로드하지 않아도 된다

단점:

- 세션/쿠키 보관 정책 검토 필요
- 원격 브라우저 보안 검토 필요
- 캡차/2차 인증이 있으면 사용자 개입 필요
- 광고센터 정책 변경에 취약

#### 방식 B: 사용자 PC 로컬 보조 프로그램

```text
사용자 PC에서 브라우저 자동화 실행
사용자 다운로드 폴더 감지
AIMAOS 서버로 파일 업로드
```

장점:

- 사용자가 자기 PC에서 직접 로그인하므로 계정 통제가 명확하다
- 다운로드 폴더 감지가 쉽다

단점:

- 설치가 필요하다
- 현재 목표인 설치 없는 웹 기반 UX와 맞지 않는다

#### 방식 C: 브라우저 확장 프로그램

```text
확장 프로그램이 광고센터 다운로드를 감지
AIMAOS로 파일 전송
```

장점:

- 다운로드 감지와 세션 활용이 자연스럽다

단점:

- 확장 프로그램 배포와 심사가 필요하다
- MVP 범위에는 무겁다

### 7.4 1차 권장 방식

1차 POC는 방식 A를 권장한다.

다만 MVP에서는 세션/쿠키 저장 없이 "사용자가 매번 직접 로그인 후 해당 세션에서 리포트 다운로드"까지 검증한다.

세션 저장은 2차 검증으로 미룬다.

## 8. 기존 AIMAOS 파이프라인 연결

반자동 또는 API 수집 결과는 기존 MVP 구조를 그대로 사용한다.

```text
수집 원본
  ↓
data/raw/{media}/{date}/ 저장
  ↓
raw_media_audit.py 진단
  ↓
column_mapper.py 매핑
  ↓
standardizer.py 표준화
  ↓
performance_analyzer.py 분석
  ↓
action_engine.py 오늘 해야 할 일 생성
  ↓
report_generator.py 보고서 생성
```

기존 분석 로직은 바꾸지 않는다. 수집 레이어는 "파일을 대신 가져오는 역할"만 한다.

## 9. 보안 원칙

| 항목 | 원칙 |
| --- | --- |
| 광고주 비밀번호 | 저장 금지 |
| API Key/Secret | 암호화 저장 |
| 세션/쿠키 | MVP에서는 저장하지 않음 |
| 캡차 | 사용자가 직접 처리 |
| 2차 인증 | 사용자가 직접 처리 |
| 서버 직접 로그인 | MVP에서 제외 |
| 감사 로그 | 누가, 언제, 어떤 계정을 수집했는지 기록 |
| 다운로드 원본 | 추적 가능하게 저장 |
| 권한 회수 | 연결 해제와 키 삭제 제공 |

암호화 저장 후보:

- 로컬 MVP: OS Credential Store 또는 `.env` + 별도 암호화 키
- SaaS: KMS 또는 Vault 계열
- DB: secret 값은 평문 저장 금지

## 11. 대행사 계정 테스트 계획

현재 사용자는 광고대행사 계정으로 여러 광고주 접근 권한이 있으므로 1차 POC는 대행사 계정 기준이 적합하다.

### 11.1 네이버

확인할 항목:

1. 대행사 계정에서 API Manager 접근 가능 여부
2. Customer ID / Access License / Secret Key 발급 가능 여부
3. API 호출 시 관리 광고주 목록 또는 특정 광고주 데이터 접근 가능 여부
4. 캠페인/광고그룹/키워드 조회 가능 여부
5. 전일 성과 통계 조회 가능 여부
6. 전환수/매출 데이터가 실제로 내려오는지 여부
7. 여러 광고주를 계정 별칭으로 구분 가능한지 여부

### 11.2 G마켓/옥션

확인할 항목:

1. 대행사 계정으로 광고센터 접근 가능 여부
2. 광고주 선택 UI 존재 여부
3. 리포트 페이지 URL 패턴
4. 기간 선택 방식
5. 리포트 종류: 일자별, 월별, 그룹별, 상품별 등
6. 다운로드 파일명 규칙
7. legacy 광고센터와 신규 광고센터 리포트 구조 차이

### 11.3 11번가

확인할 항목:

1. 대행사 계정으로 광고주 전환 가능 여부
2. 광고 리포트 다운로드 가능 여부
3. 기간 선택과 파일 형식
4. 로그인 세션 유지 방식
5. 원본 컬럼명과 AIMAOS 매핑 가능성

## 12. 일반 광고주 계정 테스트 계획

대행사 계정 POC 후 일반 광고주 계정으로 2차 검증한다.

확인할 항목:

- API Key를 광고주가 직접 발급할 수 있는가
- 광고주가 키를 입력하는 UX를 이해할 수 있는가
- 대행사 권한이 없는 단일 계정에서도 동일한 데이터가 내려오는가
- G마켓/옥션/11번가에서 광고주 직접 로그인 시 리포트 접근 경로가 동일한가
- 2차 인증/캡차 빈도가 어느 정도인가

## 13. 1차 POC 개발 범위

### 13.1 포함

| 범위 | 내용 |
| --- | --- |
| 네이버 연결 정보 입력 화면 | Customer ID, Access License, Secret Key, 계정 별칭 |
| 연결 테스트 | 캠페인 또는 계정 기본 조회 |
| 전일 성과 수집 | 1개 광고주, 1일 단위 |
| 원본 저장 | JSON/CSV 원본 저장 |
| 표준화 연결 | 기존 column_mapper/standardizer 활용 |
| 오늘 해야 할 일 연결 | 기존 action_engine 참조 |
| 보고서 생성 연결 | 기존 보고서 구조 유지 |
| 수집 로그 | 성공/실패/권한 부족 기록 |

### 13.2 제외

| 제외 항목 | 이유 |
| --- | --- |
| 자동입찰 | AIMAOS v1 범위 아님 |
| 입찰가/예산 변경 | 광고주 승인 체계 전까지 제외 |
| G마켓/옥션/11번가 완전 자동 로그인 | 비밀번호 저장과 정책 리스크 |
| 캡차 우회 | 정책 위반 가능성 |
| 세션 장기 저장 | 보안 검토 후 2차 |
| 전 매체 동시 개발 | MVP 안정성 우선 |

## 14. 구현 우선순위

### 1순위: 네이버 검색광고 API 읽기 전용 POC

가장 안정적이다. API 기반이며, 기존 AIMAOS 분석 파이프라인과 연결하기 쉽다.

목표:

- 전일 성과 자동 수집
- 오늘 해야 할 일 자동 생성
- 보고서 센터 자동 반영

### 2순위: G마켓/옥션 리포트 반자동 수집 POC

이미 legacy 리포트 구조 대응 준비가 있으므로 우선순위가 높다.

목표:

- 사용자가 직접 로그인
- 리포트 다운로드 자동화
- 다운로드 파일을 기존 진단/분석으로 연결

### 3순위: 11번가 반자동 수집 POC

원본 파일 확보와 화면 구조 확인이 선행되어야 한다.

### 4순위: 세션 저장과 스케줄링

보안 검토 후 진행한다.

## 15. 최종 판단

### 네이버

API 기반 자동 수집을 1차 POC로 진행할 가치가 높다.

이유:

- 공식 검색광고 API가 존재한다.
- 캠페인, 광고그룹, 키워드, 통계 수집 가능성이 높다.
- 기존 AIMAOS 분석 구조와 연결하기 쉽다.
- 광고주가 매일 파일을 업로드하지 않아도 된다.

### G마켓/옥션/11번가

공식 API 여부를 계속 확인하되, 1차 실무 대응은 반자동 리포트 수집이 현실적이다.

이유:

- 판매자 API와 광고 성과 리포트 API는 다를 수 있다.
- 광고센터 화면과 리포트 다운로드는 실무에서 이미 사용 중이다.
- 비밀번호 저장 없이 사용자가 직접 로그인하는 구조가 안전하다.
